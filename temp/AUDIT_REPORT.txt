§5 — ANALYSIS OF TEAMMATE'S "GLOBAL LOCK" SUGGESTION
     (Critical Concurrency Reminder)
================================================================

Your teammate proposed adding acquire_global_lock() /
release_global_lock() calls inside every main() function.

THIS SUGGESTION IS INCORRECT AND SHOULD NOT BE IMPLEMENTED.
Here is the technical reasoning:

1. WHAT THE TEAMMATE IS SOLVING:
   The concern is that flock() is released at the end of load_*(),
   so the .dat file is "unprotected" while business logic runs.
   This is a valid observation about the GAP between load and save.

2. WHY A GLOBAL LOCK IN MAIN() IS THE WRONG SOLUTION:
   Each C binary is a SEPARATE OS PROCESS, spawned independently
   by Flask's subprocess.run(). A global lock implemented as a C
   variable (e.g. a global int) lives in each process's own
   memory space. Process A's acquire_global_lock() has NO effect
   whatsoever on Process B. The "global lock" would only protect
   against concurrent calls within the SAME process, which never
   happens (each Flask request spawns a fresh process).

3. WHAT ACTUALLY PROTECTS THE DATA:
   flock() IS a cross-process mechanism — it is a kernel-level
   advisory lock on the FILE DESCRIPTOR. When save_order_sll()
   opens orders.dat and calls flock(LOCK_EX), no other process
   can acquire an exclusive lock on that file until this process
   calls flock(LOCK_UN) and closes it. This IS the correct protection.

4. THE REAL GAP (and why it is acceptable here):
   There is a window between load (LOCK_SH released) and save
   (LOCK_EX acquired) during which another process could also
   load the same file. In a high-concurrency production system,
   this would require a "lock-before-load, unlock-after-save"
   pattern. For a college demo project with a small user base
   and Flask's default single-threaded subprocess model, this
   window is acceptable. The save_*_sll() exclusive lock still
   prevents WRITE-WRITE corruption (the most dangerous case).

5. CORRECT FIX (if needed in future):
   If you want true load-to-save protection, change the load
   macros to hold LOCK_EX from open until save:

   Instead of:
     load → LOCK_SH → read → UNLOCK → (gap) → save → LOCK_EX → write → UNLOCK

   Do:
     open fd → LOCK_EX → fread into SLL → [do NOT close/unlock] →
     business logic → fwrite SLL back → UNLOCK → fclose

   This requires passing the open fd through to save, which is
   a larger refactor. For SDP-1 scope, the current design is fine.

CONCLUSION: Do NOT add acquire_global_lock() / release_global_lock()
to main(). It provides zero actual protection and adds dead code.
