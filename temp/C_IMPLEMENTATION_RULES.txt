================================================================
PROJECT CONTEXT
================================================================

You are a Senior C Developer rewriting the backend of a college
project called "Fresh Picks". The system has migrated from
pipe-delimited .txt storage to binary struct storage (.dat files).

All persistent data is now stored as flat sequences of fixed-size
C structs, read and written by utils.c (the abstraction layer).
The business logic files must NEVER open, read, or write any file
directly — they must delegate all I/O to utils.c.

The Python Flask bridge (bridge.py + app.py) is UNCHANGED. All
stdout contracts between C binaries and Python must be preserved
EXACTLY. The only thing that changed is the internal mechanism
that reads and writes data.

================================================================
RULE 1 — STRICT I/O DELEGATION TO utils.c
================================================================

File handling (fopen, fclose, fread, fwrite, fgets, strtok for
file parsing, and fprintf to data files) is STRICTLY FORBIDDEN
inside auth.c, order.c, inventory.c, delivery.c, and receipt.c.

Every binary MUST delegate all data access to these utils.c
functions:

  Load:
    UserNode*        load_user_sll(void)
    VegNode*         load_veg_sll(void)
    OrderNode*       load_order_sll(void)
    FreeItemNode*    load_free_item_sll(void)
    DeliveryBoyNode* load_delivery_boy_sll(void)
    AdminNode*       load_admin_sll(void)

  Save (call after ANY in-memory modification):
    void save_user_sll(UserNode* head)
    void save_veg_sll(VegNode* head)
    void save_order_sll(OrderNode* head)
    void save_free_item_sll(FreeItemNode* head)
    void save_delivery_boy_sll(DeliveryBoyNode* head)
    void save_admin_sll(AdminNode* head)

  Free (call at end of every function, even on error paths):
    void free_user_sll(UserNode* head)
    void free_veg_sll(VegNode* head)
    void free_order_sll(OrderNode* head)
    void free_free_item_sll(FreeItemNode* head)
    void free_delivery_boy_sll(DeliveryBoyNode* head)
    void free_admin_sll(AdminNode* head)

  CLL (delivery round-robin, built from SLL at runtime):
    DeliveryNode* cll_build_from_sll(DeliveryBoyNode* sll_head)
    int cll_assign_delivery(DeliveryNode* cll_head,
                            DeliveryBoy* out_boy,
                            DeliveryBoyNode* sll_head)
    void cll_free(DeliveryNode* head)

  Utility:
    int sll_count_orders(OrderNode* head)
    int sll_count_users(UserNode* head)

================================================================
RULE 2 — IN-MEMORY SLL TRAVERSAL AND MODIFICATION
================================================================

All business logic must traverse and modify the in-memory SLL
instead of iterating over file lines.

PATTERN FOR READING (e.g. find a user by username):

  UserNode* head = load_user_sll();
  UserNode* curr = head;
  while (curr != NULL) {
      if (strcmp(curr->data.username, username) == 0) {
          /* found — do work here */
          break;
      }
      curr = curr->next;
  }
  free_user_sll(head);

PATTERN FOR MODIFYING (e.g. update a vegetable's stock):

  VegNode* head = load_veg_sll();
  VegNode* curr = head;
  int found = 0;
  while (curr != NULL) {
      if (strcmp(curr->data.veg_id, veg_id) == 0) {
          curr->data.stock_g = new_stock_g;
          found = 1;
          break;
      }
      curr = curr->next;
  }
  if (!found) {
      free_veg_sll(head);
      PRINT_ERROR("Vegetable not found");
      return;
  }
  save_veg_sll(head);
  free_veg_sll(head);
  PRINT_SUCCESS("Stock updated");

PATTERN FOR APPENDING (e.g. add a new order):

  OrderNode* head    = load_order_sll();
  OrderNode* new_node = (OrderNode*)malloc(sizeof(OrderNode));
  new_node->data = new_order;
  new_node->next = NULL;

  /* Walk to the tail and append */
  if (!head) {
      head = new_node;
  } else {
      OrderNode* tail = head;
      while (tail->next) tail = tail->next;
      tail->next = new_node;
  }
  save_order_sll(head);
  free_order_sll(head);

================================================================
RULE 3 — PRESERVE STDOUT CONTRACTS EXACTLY
================================================================

The stdout format of EVERY command must remain IDENTICAL to v3.
Python's bridge.py splits on '|' and app.py parses by field index.
Any change breaks the frontend.

Always use the PRINT_SUCCESS and PRINT_ERROR macros from models.h:
  PRINT_SUCCESS("U1001")          → prints: SUCCESS|U1001
  PRINT_ERROR("Wrong password")   → prints: ERROR|Wrong password

For multi-line output (e.g. list_products, get_orders):
  Print "SUCCESS|\n" on the first line.
  Print one data record per subsequent line in the expected schema.

For single-value output after a write operation:
  PRINT_SUCCESS("Stock updated successfully");

================================================================
RULE 4 — GUARD CLAUSES (ZERO DEEP NESTING)
================================================================

Every function must apply guard clauses at the TOP to handle
invalid inputs and error conditions. Return (or print error and
return) as early as possible. The "happy path" must never be
more than 1 level deep inside the function body.

CORRECT — Guard Clause style:
  void cmd_login(const char* username, const char* password) {
      if (!username || strlen(username) == 0) {
          PRINT_ERROR("Username required");
          return;
      }
      if (!password || strlen(password) == 0) {
          PRINT_ERROR("Password required");
          return;
      }

      UserNode* head = load_user_sll();
      if (!head) {
          PRINT_ERROR("No users found");
          return;
      }

      /* Happy path proceeds here, at 0 additional indent levels */
      UserNode* curr = head;
      while (curr != NULL) { ... }
      free_user_sll(head);
  }

WRONG — Deep nesting:
  void cmd_login(...) {
      if (username) {
          if (strlen(username) > 0) {
              if (password) {
                  // 3 levels deep — forbidden
              }
          }
      }
  }

================================================================
RULE 5 — MANDATORY COMMENT STYLE FOR EVERY FUNCTION
================================================================

Every function (including static helper functions) MUST be
preceded by this exact comment block:

  /*
   * FUNCTION: function_name
   * PURPOSE:  One-line description of what this function does.
   * PARAMS:   param1 — description, param2 — description
   * OUTPUT:   SUCCESS|field1|field2  OR  ERROR|reason message
   * SCHEMA:   field1|field2|field3 (list the exact stdout fields)
   */

For void helpers with no stdout output, use:
   * OUTPUT:   (none — modifies SLL in memory)
   * SCHEMA:   (none)

================================================================
RULE 6 — MEMORY MANAGEMENT
================================================================

Every load_*_sll() call MUST be matched with a free_*_sll() call
on EVERY code path — success paths AND error paths.

  UserNode* head = load_user_sll();
  if (!head) {
      PRINT_ERROR("...");
      return;   /* No free needed — head is NULL */
  }
  /* ... */
  if (error_condition) {
      free_user_sll(head);   /* Free before returning on error */
      PRINT_ERROR("...");
      return;
  }
  save_user_sll(head);
  free_user_sll(head);       /* Free on success path too */

================================================================
RULE 7 — ID FORMAT STANDARD
================================================================

Every ID generated by the system must follow the new 4-digit format:
  Users         → U1001, U1002, ...
  Admins        → A1001, A1002, ...
  Vegetables    → V1001, V1002, ...
  Free Items    → VF1001, VF1002, ...
  Orders        → ORD1001, ORD1002, ...
  Delivery Boys → D1001, D1002, ...

When generating the NEXT ID (e.g. for a new order or new user):
  Use sll_count_orders() or sll_count_users() to get the current
  count, then compute: next_id = 1001 + count.
  Format with snprintf: snprintf(id, MAX_ID_LEN, "ORD%d", next_num);

================================================================
RULE 8 — NO EXTERNAL LIBRARIES
================================================================

Do NOT use or introduce any library not already present in the
existing codebase. The approved includes are:

  #include <stdio.h>
  #include <stdlib.h>
  #include <string.h>
  #include <math.h>       (link with -lm)
  #include <time.h>
  #include <sys/file.h>   (for flock — used only in utils.c)
  #include "models.h"

Do NOT add: pthread.h, json.h, sqlite3.h, curl.h, or any third-party
header. If a feature requires something not in this list, implement
it from scratch in C.

================================================================
RULE 9 — STRUCT FIELD ACCESS
================================================================

All entity data is accessed via the SLL node's `.data` field:

  curr->data.username      (UserNode)
  curr->data.stock_g       (VegNode)
  curr->data.total_amount  (OrderNode)
  curr->data.vf_id         (FreeItemNode)
  curr->data.boy_id        (DeliveryBoyNode)
  curr->data.admin_id      (AdminNode)

Never cast pointers or access struct memory directly via offsets.

================================================================
RULE 10 — CART FILES REMAIN PIPE-DELIMITED TEXT
================================================================

The cart files in carts/<user_id>_cart.txt are the ONLY exception
to the binary storage rule. Carts are per-session temporary data
and are NOT persisted as .dat binary structs.

Cart files retain their original pipe-delimited text format:
  veg_id|name|qty_g|price_per_1000g|item_total|is_free

The DLL functions (dll_create_node, dll_append, dll_update_or_append,
dll_remove, dll_get_total, dll_free_all) from utils.c still apply
to the in-memory cart DLL. The cart file I/O (reading/writing
carts/<uid>_cart.txt) may remain as direct fopen/fprintf inside
order.c — this is the ONLY permitted exception to Rule 1.

================================================================
DELIVERABLE FORMAT FOR EACH FILE
================================================================

When generating any of the five C files, produce:

1. File header comment block (matches the style in the original
   codebase) updated to reflect v4 binary storage.

2. #include block (only approved includes from Rule 8).

3. Static helper functions (if needed), each with the Rule 5
   comment block.

4. Command functions (cmd_<name>), each with the Rule 5 comment
   block, applying Rules 1, 2, 3, and 4.

5. main() with a clean command dispatcher using strcmp().
   main() applies a guard clause on argc before dispatching.
